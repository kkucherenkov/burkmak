# Docker Release Images + Homelab Deploy — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Tag-driven GitHub workflow that publishes production backend+web Docker images to GHCR, plus a `deploy/` compose stack so a homelab installs the app with `docker compose up -d`.

**Architecture:** Two new multi-stage production Dockerfiles (self-contained — no repo mount, no boot-time install). A `release.yml` workflow builds both images for amd64+arm64 on tag push `v*.*.*` and creates the GitHub Release. A `docker-build.yml` PR workflow smoke-builds the Dockerfiles so a release tag is never their first build. `deploy/` holds the homelab compose file + env template + README.

**Tech Stack:** Docker buildx + QEMU, GHCR (GitHub Packages), pnpm 10.33.0 (corepack), Prisma 7 migrate deploy, Nuxt/Nitro node server.

**Spec:** `specs/features/2026-07-14-docker-release-images.design.md`

## Global Constraints

- Images: `ghcr.io/kkucherenkov/burkmak-backend`, `ghcr.io/kkucherenkov/burkmak-web`. Tags: `X.Y.Z` (from git tag `vX.Y.Z`) + `latest`. Platforms: `linux/amd64,linux/arm64`.
- Trigger: tag push `v*.*.*` only. Registry auth: built-in `GITHUB_TOKEN` (`packages: write`) — no new secrets.
- Backend base image `node:24-slim` (glibc — better-sqlite3 prebuilds); web base image `node:24-alpine`. Both runtime stages run as the `node` user (non-root).
- pnpm is pinned by root `package.json` `"packageManager": "pnpm@10.33.0"`; activate with `corepack enable` in every image build stage.
- NEVER run `turbo run build` or `pnpm spec:codegen` inside a Docker build — the turbo graph pulls Dart codegen, which needs the Flutter toolchain. Generated clients are committed; call per-package `pnpm --filter <pkg> <script>` directly.
- Migrations run at container boot via `prisma migrate deploy` (from the image's own `node_modules/.bin/prisma`, offline). `prisma db push` remains prohibited.
- `pnpm deploy` needs the `--legacy` flag on pnpm 10 (verified working: `pnpm --filter @app/backend deploy --prod --legacy <dir>` exits 0 and compiles the better-sqlite3 binding via root `onlyBuiltDependencies`).
- Backend env contract (all existing `AppConfig` keys, no code changes): `DATABASE_URL`, `DATA_DIR`, `PORT`, `BETTER_AUTH_SECRET`, `BETTER_AUTH_URL`, `BETTER_AUTH_BASE_PATH`, `CORS_ORIGINS`. Web env contract: `NUXT_PUBLIC_API_BASE_URL`, `NUXT_PUBLIC_AUTH_BASE_URL`, `PORT`.
- The host dev stack (`docker/compose.yml`) keeps ports 3000/3001 busy — every local container test in this plan uses other host ports (3100/3101/3300/3301) and, for compose, a separate project name (`-p burkmak-deploy-test`).
- The repo `.dockerignore` already excludes `node_modules`, `dist`, `.output`, `*.db` — do not weaken it; production builds create all artifacts inside the image.
- After editing `.md`/`.json`/`.yml` files run `pnpm format` (Prettier) before committing.

- [ ] **Step 4: Create `apps/backend/Dockerfile`**

```dockerfile
# syntax=docker/dockerfile:1

# ── build ────────────────────────────────────────────────────────────────────
FROM node:24-slim AS build

# Toolchain is a fallback: better-sqlite3 normally downloads a glibc prebuild,
# but if one is missing for a platform, node-gyp compiles from source.
RUN apt-get update \
  && apt-get install -y --no-install-recommends python3 make g++ \
  && rm -rf /var/lib/apt/lists/*
RUN corepack enable

WORKDIR /repo
ENV CI=1

COPY . .

RUN pnpm install --frozen-lockfile
RUN pnpm --filter @app/specs bundle
RUN pnpm --filter @app/backend prisma:generate
RUN pnpm --filter @app/backend build

# Self-contained production dir: backend files + pruned prod node_modules.
# --legacy is required on pnpm 10 (new deploy mode needs injected workspaces).
RUN pnpm --filter @app/backend deploy --prod --legacy /prod
# deploy selects files npm-pack-style, which respects .gitignore — the
# gitignored build artifact dist/ must be copied in explicitly.
RUN cp -a apps/backend/dist /prod/dist && test -f /prod/dist/main.js
# The deploy re-installs node_modules, so the Prisma client must be
# regenerated against them (CLI is a prod dependency).
RUN cd /prod && node_modules/.bin/prisma generate
# Source and tests are not needed at runtime — dist/ carries the app.
RUN rm -rf /prod/src /prod/test /prod/Dockerfile.dev

# ── runtime ──────────────────────────────────────────────────────────────────
FROM node:24-slim AS runtime

ENV NODE_ENV=production \
    PORT=3000

WORKDIR /app
COPY --from=build --chown=node:node /prod /app
# COPY --chown only chowns the files/dirs it copies in, not the /app
# directory entry itself (WORKDIR already created it as root:root). Without
# this, the non-root user below can write into existing subpaths but not
# create new top-level ones (e.g. the mock storage service's ./.storage).
RUN chown node:node /app

# SQLite + extracted-article caches live here; pre-owning the mount point
# makes the named volume writable for the non-root user on first use.
RUN mkdir /data && chown node:node /data

USER node
EXPOSE 3000

HEALTHCHECK --interval=30s --timeout=5s --start-period=60s --retries=3 \
  CMD node -e "fetch('http://127.0.0.1:'+(process.env.PORT||3000)+'/api/v1/health').then(r=>process.exit([200,503].includes(r.status)?0:1)).catch(()=>process.exit(1))"

CMD ["sh", "-c", "node_modules/.bin/prisma migrate deploy && node dist/main.js"]
```

```dockerfile
# syntax=docker/dockerfile:1

# ── build ────────────────────────────────────────────────────────────────────
FROM node:24-alpine AS build

RUN apk add --no-cache libc6-compat && corepack enable

WORKDIR /repo
ENV CI=1 \
    NUXT_TELEMETRY_DISABLED=1

COPY . .

RUN pnpm install --frozen-lockfile
RUN pnpm --filter @app/specs bundle
RUN pnpm --filter @app/design-tokens build
RUN pnpm --filter @app/ui build
RUN pnpm --filter @app/web build

# ── runtime ──────────────────────────────────────────────────────────────────
FROM node:24-alpine AS runtime

ENV NODE_ENV=production \
    PORT=3001 \
    NUXT_TELEMETRY_DISABLED=1

WORKDIR /app
# Nitro bundles its own dependencies — .output is the entire app.
COPY --from=build --chown=node:node /repo/apps/web/.output ./.output

USER node
EXPOSE 3001

HEALTHCHECK --interval=30s --timeout=5s --start-period=15s --retries=3 \
  CMD node -e "fetch('http://127.0.0.1:'+(process.env.PORT||3001)+'/').then(r=>process.exit(r.status===200?0:1)).catch(()=>process.exit(1))"

CMD ["node", ".output/server/index.mjs"]
```

# Deploy burkmak on your homelab

Two containers (API + web app) and one data volume. Images are published to
GitHub Packages on every release:
[`burkmak-backend`](https://github.com/kkucherenkov/burkmak/pkgs/container/burkmak-backend) ·

## Install

```sh
mkdir burkmak && cd burkmak
curl -fsSLO https://raw.githubusercontent.com/kkucherenkov/burkmak/main/deploy/compose.yml
curl -fsSL https://raw.githubusercontent.com/kkucherenkov/burkmak/main/deploy/.env.example -o .env
$EDITOR .env        # set BURKMAK_HOST and BETTER_AUTH_SECRET
docker compose up -d
```

Open `http://<BURKMAK_HOST>:3001`, create your account, start saving articles.

## Upgrade

```sh
docker compose pull && docker compose up -d
```

Pin a specific version instead of `latest` by setting `IMAGE_TAG=0.1.0` in
`.env`.

## Backup

Everything lives in the `burkmak-data` volume (SQLite database + cached
articles, images, and EPUBs):

```sh
docker run --rm -v burkmak_burkmak-data:/data -v "$PWD":/backup alpine \
  tar czf /backup/burkmak-backup.tar.gz -C /data .
```

## Limitations over plain HTTP

This stack serves plain HTTP on your LAN. The web app and OPDS downloads
work fine, but **native Kobo store sync requires HTTPS** — Kobo firmware
refuses http:// endpoints. Put the backend behind your own TLS reverse
proxy (and set `BETTER_AUTH_URL`/`CORS_ORIGINS` accordingly) if you want
device sync.

### Task 4: Release + PR smoke-build workflows

**Files:**

- Create: `.github/workflows/release.yml`
- Create: `.github/workflows/docker-build.yml`

**Interfaces:**

- Consumes: `apps/backend/Dockerfile`, `apps/web/Dockerfile` (Tasks 1–2), image names from Global Constraints.
- Produces: on tag `vX.Y.Z` — GHCR packages `burkmak-backend`/`burkmak-web` tagged `X.Y.Z` + `latest` and a GitHub Release; on PRs touching docker files (and on push to `main`) — an amd64 smoke build using the same buildx cache scope as releases, warmed by the post-merge `main` build since PR-ref caches are branch-isolated and unreadable from tag pushes.

- [ ] **Step 1: Create `.github/workflows/release.yml`**

```yaml
name: Release

on:
  push:
    tags: ['v*.*.*']

permissions:
  contents: write
  packages: write

jobs:
  build-images:
    name: Build & push ${{ matrix.image }}
    runs-on: ubuntu-latest
    strategy:
      matrix:
        include:
          - image: burkmak-backend
            dockerfile: apps/backend/Dockerfile
          - image: burkmak-web
            dockerfile: apps/web/Dockerfile
    steps:
      - uses: actions/checkout@v6

      - uses: docker/setup-qemu-action@v3

      - uses: docker/setup-buildx-action@v3

      - uses: docker/login-action@v3
        with:
          registry: ghcr.io
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}

      - id: meta
        uses: docker/metadata-action@v5
        with:
          images: ghcr.io/${{ github.repository_owner }}/${{ matrix.image }}
          # Semver tag from the git tag; the default flavor also adds
          # `latest` for non-prerelease versions.
          tags: |
            type=semver,pattern={{version}}

      - uses: docker/build-push-action@v6
        with:
          context: .
          file: ${{ matrix.dockerfile }}
          platforms: linux/amd64,linux/arm64
          push: true
          tags: ${{ steps.meta.outputs.tags }}
          labels: ${{ steps.meta.outputs.labels }}
          cache-from: type=gha,scope=${{ matrix.image }}
          cache-to: type=gha,mode=max,scope=${{ matrix.image }}

  github-release:
    name: Create GitHub Release
    needs: build-images
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v6

      - name: Create release with generated notes
        env:
          GH_TOKEN: ${{ secrets.GITHUB_TOKEN }}
        run: |
          gh release create "$GITHUB_REF_NAME" \
            --title "$GITHUB_REF_NAME" \
            --generate-notes \
            --notes "Docker images (linux/amd64 + linux/arm64):
          - \`ghcr.io/kkucherenkov/burkmak-backend:${GITHUB_REF_NAME#v}\`
          - \`ghcr.io/kkucherenkov/burkmak-web:${GITHUB_REF_NAME#v}\`

          Homelab install guide: [deploy/README.md](https://github.com/kkucherenkov/burkmak/blob/main/deploy/README.md)"
```

## Post-merge release rehearsal (not a plan task — controller/user runs after the PR merges)

1. `git tag v0.1.0 && git push origin v0.1.0`
2. Watch the Release run: both images push for both arches; the GitHub Release appears.
3. Verify: `docker manifest inspect ghcr.io/kkucherenkov/burkmak-backend:0.1.0 | grep -E 'amd64|arm64'` shows both platforms; same for `burkmak-web`; `latest` resolves to the same digests.
4. The `docker-build.yml` check will already have validated both Dockerfiles on the PR itself (the PR touches them, matching its own path filter).
