## Goal

Read your saved articles on a Kobo e-reader. burkmak generates an **EPUB/KEPUB**
from each extracted article and serves an **OPDS catalog** the Kobo subscribes to
(Settings → Add OPDS catalog), so new articles appear on-device over wifi and
download in one tap — no USB, no firmware hacking.

At the end of this slice a user can: create a PAT, add `https://<host>/api/v1/opds`
as a custom OPDS catalog on their Kobo (or any OPDS reader) authenticating with
the token, browse their extracted articles, and download each as a Kobo-friendly
`.kepub.epub`.

## Scope decision (informed by BookLore / Calibre-Web / Komga)

Those projects implement **native Kobo store-API emulation**: the user overrides
`api_endpoint` in `Kobo eReader.conf` to a token-in-URL (`…/kobo/<token>`), and
the server emulates ~8–10 `/v1/*` endpoints (`/v1/initialization`,
`/v1/library/sync`, `…/metadata`, `…/state` GET+PUT, …), runs a **delta-sync
state machine** (per-user library snapshot + entitlement UUIDs/revisions to emit
New/Changed/Deleted), proxies unhandled calls to `storeapi.kobo.com`, and accepts
**reading-progress sync-back**. It buys "appears automatically" UX + bidirectional
progress — at the cost of a stateful delta engine, a reverse-proxy/TLS hard
dependency, and Nickel-firmware fragility whose failures are **device-emitted and
unverifiable without a physical Kobo**.

**This slice ships OPDS + EPUB/KEPUB**, which a stock Kobo consumes natively via
its built-in custom-OPDS-catalog feature — small surface, standards-based, fully
testable here in any OPDS client. **Native sync is a documented fast-follow** (see
below).

**In:**

- EPUB3 builder from an extracted `Article` (+ cached images).
- Optional KEPUB span injection (`.kepub.epub`) for accurate on-device progress.
- `GET /api/v1/items/{id}/epub` — on-demand build, disk-cached, streamed.
- `GET /api/v1/opds` — OPDS 1.2 Atom acquisition feed of extracted articles.
- PAT auth (HTTP Basic, password = token) on both.

**Out (deliberately, → fast-follow slice `kobo-native-sync`):**

- Native Kobo store-API emulation (`/v1/initialization`, `/v1/library/sync`,
  metadata/state, device-auth stub, store catch-all proxy).
- The entitlement/revision **delta-sync state machine** + per-device snapshots.
- **Read-state sync-back** from device → burkmak.
- These need a physical Kobo to verify; reference impls: BookLore
  `KoboController`/`KoboLibrarySyncService`, Calibre-Web `cps/kobo.py`.

## Architecture

### EPUB builder (pure, `jszip`)

`buildEpub(article, item, images): Buffer/Uint8Array` → a valid EPUB3 zip:

- `mimetype` **stored first, uncompressed** (EPUB spec requirement).
- `META-INF/container.xml` → points at `OEBPS/content.opf`.
- `OEBPS/content.opf` — metadata (title, language, identifier =
  `urn:burkmak:<itemId>`, author/`dc:source` = `siteName`/`url`, `dcterms:modified`),
  manifest (chapter xhtml + nav + each embedded image), spine.
- `OEBPS/nav.xhtml` — EPUB3 nav (single chapter).
- `OEBPS/chapter.xhtml` — the sanitized `contentHtml` wrapped in valid XHTML, with
  `<img src>` rewritten from `/api/v1/items/{id}/image/{key}` to the **local
  packaged** image path (`images/<key>.<ext>`), reading bytes from the existing
  on-disk image cache.
- Images embedded from `apps/backend/data/images/<itemId>/…`.

Pure + deterministic → unit-test by asserting zip entries, `mimetype` is first &
stored, container/opf reference the chapter + images, XHTML is well-formed.

**KEPUB variant:** `toKepub(epubXhtml)` wraps text runs in
`<span class="koboSpan" id="kobo.<para>.<seg>">…</span>` (Kobo Nickel uses these
for progress %, time-remaining, stats). Pure string transform → unit-test spans
present, text preserved, existing markup not double-wrapped. Served when the
acquisition/download path requests KEPUB (filename `*.kepub.epub`). No external
`kepubify` binary (its download would stall in this sandbox).

### Download endpoint

`GET /api/v1/items/{id}/epub` (PAT or session): ownership check → require
`extractStatus = ready` (else `409`) → build (or read disk cache keyed by
`itemId + article.extractedAt + variant`, mirroring the image cache) → stream
`application/epub+zip` with `Content-Disposition: attachment; filename="<slug>.kepub.epub"`,
`X-Content-Type-Options: nosniff`. Reuses the existing image-route streaming
pattern.

### OPDS catalog

`GET /api/v1/opds` (PAT via Basic): an **OPDS 1.2 acquisition feed** (Atom XML,
`application/atom+xml;profile=opds-catalog;kind=acquisition`) listing the user's
extracted articles (newest first; default unread+read, exclude archived), each
`<entry>` with title, author (`siteName`), summary (`excerpt`), `updated`, and an
acquisition `<link rel="http://opds-spec.org/acquisition" type="application/epub+zip"
href="…/items/{id}/epub">`. A pure `buildOpdsFeed(items, baseUrl): string` →
unit-test valid Atom, entry count, acquisition hrefs, XML-escaping of titles.

(OPDS is XML, not JSON — documented in `openapi.yaml` as a path with
`content: application/atom+xml`; no client codegen consumes it.)

## API surface (spec-first → codegen)

New tag `kobo`. Paths: `GET /api/v1/items/{id}/epub` (binary, like the image
route) and `GET /api/v1/opds` (atom-xml). No new JSON schemas needed; both are
non-JSON responses. No new DB model (reads `Item`/`Article`; EPUB disk-cached
like images under `apps/backend/data/epub/`, gitignored).

## Error handling

- Item not owned → `404`. Item without a ready article → `409` (nothing to
  package). Missing cached image → skip that image (don't fail the book).
- OPDS with no extracted articles → a valid empty feed (not an error).

## Testing (DoD)

- EPUB builder units (zip structure, mimetype-first, opf/manifest, image
  embedding, XHTML validity); KEPUB span-injection units.
- OPDS feed-builder units (Atom validity, entries, acquisition links, escaping).
- Endpoint/integration: `/items/{id}/epub` 200 streams epub for a ready item,
  409 for not-ready, 404 cross-user; `/opds` 200 atom, PAT-Basic accepted /
  rejected.
- Backend gates: `pnpm --filter @app/backend {lint,typecheck,test}`.

## Verification ceiling

EPUB/KEPUB bytes, OPDS XML, and auth are **fully testable here** (validate the
EPUB in an EPUB validator / open in Foliate/Thorium; OPDS in any OPDS reader).
The **on-device Kobo experience** (catalog discovery, KEPUB progress) needs a
physical Kobo — out of sandbox; documented, like S3-mobile's ceiling.

## Follow-on slice (not in this build): `kobo-native-sync`

Native store-API emulation + delta-sync + read-state sync-back (BookLore /
Calibre-Web design). Adds: a `KoboDevice`/snapshot + reading-state model, the
`/v1/*` endpoint set, the entitlement-delta engine, and a `storeapi.kobo.com`
catch-all proxy. **Requires a physical Kobo to verify** — budget a device before
shipping it.
