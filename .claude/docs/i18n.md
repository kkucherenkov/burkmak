# Internationalisation

No user-visible string may ship as a literal. Ever. Three platforms, three implementations — same discipline.

Default locale: `en`. Every key in `en` **must** exist in `ru` (and `el`, `uk` where added). Missing translation is a bug, not a fallback.

## Web — `@nuxtjs/i18n` + global `.ts` langDir locale files

> WHY: Vite 8 + `@nuxtjs/i18n` cannot build SFC `<i18n lang="json">` blocks —
> they produce `?vue&type=i18n&lang.json` virtual modules that Vite 8 refuses
> to parse at build time. `.ts` locale files avoid the virtual-module path
> entirely and are the supported approach with this toolchain.

Translations live in **two global files** under `apps/web/i18n/locales/`:

```
apps/web/i18n/
  i18n.config.ts          ← defineI18nConfig({ legacy: false })
  locales/
    en.ts                 ← export default { <namespaces> } as const
    ru.ts                 ← same shape, Russian values
```

Each file exports a single nested object whose top-level keys are **namespaces** matching the page/component (e.g. `welcome`, `signIn`, `library`, `nav`):

```ts
// apps/web/i18n/locales/en.ts
export default {
  welcome: {
    title: 'Your quiet reading room',
    // ...
  },
  signIn: {
    title: 'Welcome back',
    // ...
  },
  // ...
} as const;
```

In every page/component, call `useI18n()` at **global scope** (no `useScope: 'local'`) and prefix every key with the namespace:

```ts
// <script setup>
const { t } = useI18n();
```

```html
{{ t('welcome.title') }} {{ t('signIn.errorCredentials') }} {{ t('library.seg.unread') }}
```

Rules:

- `en.ts` and `ru.ts` **must have identical key structure** — missing key is a bug.
- `@app/ui` components **never** call `useI18n()` — receive translated strings via props.
- To embed a literal `@`, escape: `{'@'}` (vue-i18n linked-message syntax).
- `nuxt.config.ts` registers each locale with `file: 'en.ts'` / `file: 'ru.ts'`. Do NOT add the `@intlify/unplugin-vue-i18n` Vite plugin — it double-registers and conflicts.

Namespaces in use: `welcome`, `signIn`, `signUp`, `library`, `settings`, `itemDetail`, `addBar`, `addModal`, `nav`.

## Mobile — `slang` with namespace files

Files: `lib/i18n/<namespace>_<locale>.i18n.json` — one file per `(namespace, locale)` pair.

Config (`slang.yaml`): `namespaces: true`, `string_interpolation: braces`, `enum_name: AppLocale`, `flutter_integration: true`.

Regenerate: `dart run slang` (or via `build_runner`). Output `strings.g.dart` + `strings_<locale>.g.dart` — never edit by hand, gitignored.

```dart
// main()
LocaleSettings.useDeviceLocale();
runApp(TranslationProvider(child: const ProviderScope(child: MyApp())));
```

`MaterialApp` gets `locale: TranslationProvider.of(context).flutterLocale` + `supportedLocales: AppLocaleUtils.supportedLocales`.

Consume: `context.t.<namespace>.<key>` — e.g. `context.t.common.appTitle`. No raw `Text('Overall')` — review blocker.

## Backend — `nestjs-i18n` with per-locale namespace files

Files: `apps/backend/src/i18n/<locale>/<namespace>.json`. One subdirectory per language, one file per module. Copied to `dist/i18n/` via `nest-cli.json` assets rule.

`I18nModule.forRoot` in `app.module.ts`: `I18nJsonLoader`, `fallbackLanguage: 'en'`, resolvers: `QueryResolver(['lang'])`, `HeaderResolver(['x-lang'])`, `AcceptLanguageResolver`.

```ts
const lang = I18nContext.current()?.lang;
throw new UnauthorizedException(
  this.i18n.t('realtime.signInRequired', lang ? { lang } : undefined),
);
```

Translation key: `<namespace>.<key>` (filename = namespace prefix).

**Error bodies** use RFC 9457 `application/problem+json` — `title` and `detail` stay English-canonical. `nestjs-i18n` is for server-generated strings clients pass through verbatim.
